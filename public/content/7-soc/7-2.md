## 7.2 Fallback Behaviour Definition

Step 7.1 defines when autonomy is allowed.  
Step 7.2 defines what happens when it is not.

Every Safe Operating Concept (SOC) condition is linked to a **fallback behaviour**.

---

### Purpose

Ensure that every violation of an SOC condition produces a defined, safe system response.

If a condition holds → autonomy continues  
If a condition fails → the system transitions to a safe state  

---

### Approach

Each SOC condition is mapped to:

> **SOC Condition → Failure Trigger → MRS Policy → State Transition**

Where:

- **SOC Condition** — defines when autonomy is valid  
- **Failure Trigger** — identifies when that condition is violated  
- **MRS Policy** — defines the required safe response  
- **State Transition** — implements that response in the system  

This creates a direct link from **operating conditions → hazards → system behaviour**.

---

### Example: Signal Approach (SVB)

From Step 7.1:

- Autonomy during signal approach depends on the Signal Visibility Budget (SVB)

If the SVB condition fails:

- The train cannot guarantee it can stop safely  
- This corresponds to a SPAD-related failure  

**Mapping**

- SOC Condition: SVB must hold  
- Failure Trigger: braking or sighting insufficient  
- MRS Policy: immediate braking  
- State Transition: Cruise → Brake  

**Result**

> If the system cannot guarantee stopping within the available distance, it must immediately transition to braking.

---

### Example: Adhesion Uncertainty

From earlier steps:

- Braking depends on a valid lower bound on adhesion  

If adhesion cannot be reliably estimated:

**Mapping**

- SOC Condition: adhesion bound must be valid  
- Failure Trigger: adhesion uncertainty or degraded braking response  
- MRS Policy: conservative operation  
- State Transition: Nominal → Degraded / Reduced Speed  

**Result**

> If braking capability is uncertain, the system must reduce speed or enter a degraded state.

---

### What This Produces

Every SOC condition becomes **operationally enforceable**:

- no condition exists without a response  
- no failure occurs without a defined behaviour  

If a condition cannot be mapped to a fallback:

→ the Failure Management (Step 5) is incomplete and you must iterate back.
