
Deliverable D15 is produced in Step 4 and classifies the **critical event classes** that affect the validity of the admissibility conditions defined in D14.

Rather than attempting to enumerate every possible anomaly, D15 identifies **representative high-severity event classes** and defines how they interact with the system’s safety constraints.

---

## Purpose

D15 determines which events:

- **Invalidate** admissibility conditions entirely  
- **Tighten** the allowable operating envelope  
- **Render certain assumptions inapplicable**  

In doing so, it makes explicit where the **nominal safety argument no longer holds**.

This is essential for:
- Declaring **degraded behaviour** in Step 5  
- Assessing **state-of-the-art feasibility** in Step 6  

---

## Key Characteristics

Each critical event class is defined by:

- **Event Class**  
  A representative high-level category (e.g. fire, intrusion, communication failure)

- **Description**  
  The nature of the event and how it manifests operationally  

- **Invalidation Type**  
  How the event affects the admissibility predicates (invalidate, constrain, override)

- **ODM Interaction**  
  Which parts of the environment or system model are impacted  

- **Step 6 Burden**  
  What must be demonstrated or verified for the system to handle the event  

D15 focuses on **classes of events**, not individual instances, ensuring the analysis remains tractable while still covering critical failure modes.

---

## Role in the Methodology

D15 sits at the boundary between **nominal operation (Step 4)** and **failure handling (Step 5)**.

It provides:
- The conditions under which **normal operation is no longer valid**  
- The triggers for **state transitions and fallback strategies**  
- The scope for evaluating whether such handling is **feasible with current technology**  

---

## Example

Typical critical event classes might include:

- **E01: Fire onboard**  
  - Invalidation Type: Full invalidation of nominal operating envelope  
  - Effect: Immediate transition to emergency handling; standard metrics no longer apply  

- **E02: Passenger communication (alarm / intervention)**  
  - Invalidation Type: Constraint / override  
  - Effect: Alters authority assumptions and requires procedural response  

- **E03: Human intrusion on track**  
  - Invalidation Type: Conditional tightening  
  - Effect: Reduces available reaction time and may violate perception-based assumptions  

In each case, D15 defines how the event interacts with the system’s admissibility conditions and what this implies for safe operation.
